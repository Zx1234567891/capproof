## 12.2 What CapProof is — and the careful version of the gap

CapProof's contribution is **capability-consuming authorization**: high-impact actions must, **before execution, consume a scoped non-forgeable capability**, and a **deterministic reference monitor** must verify a **machine-checkable authorization chain** rooted in trusted intent. It is **not** proof-carrying actions (PCAA), **not** action-certificate governance (PCAA), **not** argument-level provenance (PACT), **not** authorization-graph detection (AUTHGRAPH), **not** trace-graph type-checking (AgentArmor), **not** a tool-call boundary rule set (CLAWGUARD), **not** value-level data-flow capabilities (CaMeL), and **not** trusted/untrusted privilege separation (PFI).

We state the gap **conservatively**, because several adjacent systems already handle pieces of it (DRIFT isolates memory; CLAWGUARD covers MCP/skill; PFI models trusted/untrusted agent flow):

> **Existing defenses can often detect or block unsafe actions. What they do not primarily do is model *authority itself* as a consumable resource — with explicit lifetime, scope, delegation attenuation, endorsement semantics, persistence restrictions, and spend state — across memory, delegation, and endorsement boundaries.** CapProof makes authority a first-class, *spendable* object: a high-impact action must present and consume a capability whose chain terminates in a trusted root, so that replay, reuse, widening, cross-agent transfer, and memory-persisted authority are all expressible (and deniable) in one mechanism.

This is a claim about **where authority is represented and enforced**, not a claim that prior systems "only see one trajectory." The kill test (`KILL_TEST_PLAN.md`) exists to check whether this representational difference yields a *real* security, usability, or auditability gap, or whether an adjacent system already closes it.

## 12.4 Why "proof-carrying" is not our headline (PCAA)

PCAA already carries action certificates / proof bundles with receipts and runtime-neutral governance. We therefore **do not claim proof-carrying as the contribution**. CapProof's identity is **capability-consuming authorization** (high-impact actions consume a scoped capability; a deterministic verifier checks the authorization chain). Whether CapProof should carry an explicit proof object *at all* is decided empirically by the **ProofObjectRemoved** ablation (§09): if removing the carried proof DAG and letting the monitor search state costs nothing in security and lowers latency, **the proof object is not the identity** — the main line stays *capability-consuming authorization*, and the proof object's value (if any) is confined to **auditability, replayability, failure localization, and decoupled construction**. CapProof and PCAA are **complementary**.

## 12.5 Conservative novelty statement

CapProof's defensible novelty is the **conjunction**, none of whose parts we claim alone: (1) **consumable (linear) action-authority capabilities** for authority-bearing arguments — *spending*, with explicit lifetime/scope/spend state, which CaMeL/PACT/AUTHGRAPH/CLAWGUARD/PFI/AgentArmor do not center; (2) a **`NO-MINT-UNTRUSTED` authority-provenance calculus** with soundness stated **relative to an explicit AuthSpec** (Oracle vs Deployed, §05); (3) explicit **delegation attenuation**, **one-shot endorsement**, and **memory-authority persistence restriction** as authority state; (4) **AuthLaunderBench** with **observable-side-effect** oracles and a **CaMeL head-to-head kill test** that checks whether (1)–(3) produce a real security/usability/auditability gap. If the kill test shows an adjacent system already covers memory/delegation/endorsement at comparable utility, the contribution narrows to (1)+(2) (still distinct), per the pivot in `KILL_TEST_PLAN.md`. We do not claim to be first to do capability, provenance, authorization graphs, or tool boundaries.
