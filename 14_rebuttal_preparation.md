# 14. Rebuttal Preparation

Fifteen anticipated questions, each with a *strong* answer and the *experiment* that backs it. Rule: never answer "did you measure X?" with prose alone — point at a number. Prior systems are named per their real mechanism (§12).

### Q1. "Is this just PCAA?"
**No.** PCAA owns proof-carrying actions / action certificates / receipts / governance, and we **retire "proof-carrying" as a headline** and cite PCAA for it. CapProof's unit is different: high-impact actions must **consume a linear capability** rooted in trusted intent under `NO-MINT-UNTRUSTED`, with **memory-authority stripping, delegation attenuation, and one-shot endorsement**. PCAA governs *what was authorized and how to replay/audit it*; it does not prevent authority from being **laundered across memory/agents/approvals**. The two are **complementary** — CapProof can emit PCAA-style receipts. **Backing:** §12.4; `ProofObjectRemoved` (§09.4) decides whether we carry a proof at all.

### Q2. "Is this just PACT plus capabilities?"
**No.** PACT does argument-level provenance with typed authority-bearing arguments (target/command/credential/content/selector/control) and isolates the LLM reasoning bottleneck. Provenance answers "where did this argument come from"; CapProof adds **consumption** (the high-impact action *spends* a token; replay is a spent-token error), plus the dimensions argument-level provenance does not, by itself, represent as consumable authority: **cross-turn memory** persistence, **cross-agent** transfer, and **approval reuse**. **Backing:** PACT-oracle/auto vs CapProof-oracle/auto on the same provenance footing (§09.2); ablations NoCapConsume/NoMemoryStrip/NoDelegCert/NoOneShot isolate each delta.

### Q3. "Is this just AUTHGRAPH with tokens?"
**No.** AUTHGRAPH aligns a clean Authorization Graph with an Injected Reasoning Graph to **detect** tool-level and parameter-source-level misalignment. CapProof's difference is not "AUTHGRAPH plus tokens": it makes authority a **consumable resource with explicit lifetime, scope, delegation attenuation, endorsement, persistence restriction, and spend state**, enforced fail-closed before execution. Detection of misalignment and *spending a rooted capability* are different objects; bolting tokens onto a detector would not give it one-shot-endorsement reuse-prevention, delegation attenuation certificates, or memory-authority persistence restrictions. On argument/parameter-source cases we expect **parity** with AUTHGRAPH, not dominance. **Backing:** AUTHGRAPH-style baseline (clean graph + tool + parameter-source alignment) vs CapProof on the laundering suites (§09, kill test).

### Q4. "Why does CaMeL not solve this?"
CaMeL's capability metadata tags **data-flow on values** (P-LLM/Q-LLM isolation + custom-interpreter data-flow graph + policies) and is **stronger than CapProof on exfiltration** — which is why we **adopt** it for G4. But its capabilities are not *spent* on a high-impact action, and value-level data-flow does not model authority **persisted in memory across turns**, **transferred between agents** by relay, or **carried over from a prior approval**. **Backing:** the CaMeL head-to-head kill test (`KILL_TEST_PLAN.md`) with pre-registered go/no-go: if CaMeL blocks the cross-boundary cases at similar utility we narrow the claim; if it over-asks we pivot to usability; if it mis-allows on memory/delegation/endorsement we have a security delta.

### Q5. "Why not CLAWGUARD rules?"
CLAWGUARD is a **strong** deterministic tool-call-boundary defense that explicitly covers content injection, MCP poisoning, and skill injection — so on KE-2/KE-5/KE-6 we expect it to **block the unsafe effect too**, and we **do not claim CapProof significantly out-ASRs CLAWGUARD on MCP/skill.** The CapProof *distinction* there is not raw blocking but **proof semantics, one-shot consumable endorsement, capability-replay prevention, delegation/endorsement scoping, failure localization, and audit/replay**: a boundary rule does not represent *whose* authority a call carries, whether *this* send is a replay of a spent token, or how to attenuate a delegated scope. **Backing:** CLAWGUARD-style baseline (boundary + approval gate, approval **not** a one-shot capability) vs CapProof on endorsement-reuse (KE-7) and delegation (KE-4); on MCP/skill we report the usability/auditability gap, not an ASR win.

### Q6. "What if the AuthSpec is wrong?"
The AuthSpec Builder is **untrusted and outside the TCB**. Theorems hold in **Oracle-AuthSpec Mode** (`G*`), so enforcement soundness does not assume the Builder is correct; in **Deployed-AuthSpec Mode** the guarantee is explicitly **relative to `G_sys`**, and the distance from true intent is **measured** as AuthSpec faithfulness. **Backing:** `AUTHSPEC_FAITHFULNESS_GATE.md` — Over/Under-Broadening Rate, High-Impact Grant Precision/Recall, Endorsement Trigger Accuracy, with §3.11 ON/OFF and the ambiguity adversary; severity threshold: high-impact over-broadening > 5–10% gates submission.

### Q7. "What if provenance is wrong?"
Provenance fidelity is assumption A1, and we **measure** exposure to it. **Backing:** PACT/CapProof oracle-vs-auto gap + injected provenance error at {1,5,10}% with an ASR-vs-error curve (§08). If the curve is a cliff we say so; the honest framing is "sound given provenance of quality q, and here is how ASR degrades below q."

### Q8. "What if an adapter misses a field?"
Then that field is in `Residual(t)` and CapProof provides **no guarantee** for effects expressible only through it — we **name** the field rather than claim completeness (residual characterization, not a probability bound). **Backing:** `TCB_AND_ADAPTER_COVERAGE.md` — Declared/Observable Authority Field Coverage, Unmodeled Side-Effect Rate, Contract Completeness, Canonicalization Coverage on a real-tool corpus; fail-closed on observable-but-undeclared fields; the bcc/headers/env/cwd/stdin/redirect residuals are expected findings.

### Q9. "How do you avoid endorsement fatigue?"
Most authority comes from the initial AuthSpec; endorsement is the exception path for `inferred`/novel high-impact bindings, is **one-shot and canonical-bound** (never silently reused or widened), and we **measure** endorsements/task (target ≤ 0.5 on static tasks) and **Endorsement Recovery Rate** (does a single ask complete the legitimate task?). **Backing:** endorsement-count distribution static vs dynamic (§08, `AUTHSPEC_FAITHFULNESS_GATE.md`); ablation NoOneShot shows removing the discipline (reuse) is unsafe.

### Q10. "Does memory stripping break utility?"
Memory still stores **content/facts**; it only refuses to let untrusted-origin values act as **authority** for high-impact arguments. Benign tasks that *recall information* are unaffected; only tasks that try to turn a remembered value into a high-impact recipient/path/endpoint trigger an endorsement. **Backing:** benign-variant Success Rate on the Memory suite and the over-block rate (§08, §10); ablation NoMemoryStrip shows the security cost of *not* stripping.

### Q11. "Does the proof object actually help?"
That is an empirical question, pre-committed to data. **Backing:** `ProofObjectRemoved` (§09.4) compares carried-proof vs monitor-search on **security (expect unchanged), latency (deep vs shallow), explainability, failure localization, auditability, replayability**. If only security matters and the rest is negligible, we **drop the proof framing** and cite PCAA; if deep-action/audit/localization benefits are material, we keep it positioned relative to PCAA.

### Q12. "Does the benchmark favor CapProof?"
We engineered against it. Oracles are **mechanism-agnostic** (observable side effects only; never AuthSpec/capability/proof); `ground_truth_authspec` is used only for CapProof-internal scoring; adaptive attacks are **held-out**; and `07`'s **threat sourcing table** ties each suite to a documented external threat with `threat_source` per task and **equal weighting** across channels (including the argument channel where we expect only parity). **Backing:** §07 oracle design + threat sourcing; held-out authorship (§09.6); per-role (not just per-channel) reporting so a reviewer can re-weight.

### Q13. "Can attackers forge a proof?"
A natural-language "proof" has no store entry. Capabilities are unforgeable handles (MAC/lookup); a fake `cap_id` fails; an LLM-judge would be the weak point, which is exactly why the monitor is **deterministic** (ablation #9 shows an LLM verifier admits fake proofs). **Backing:** adaptive attacks #1 (fake proof injection), #2 (cap forgery) → `ProofNotCanonical`/`NoCap`.

### Q14. "Can capabilities be replayed?"
No — high-impact capabilities are **linear**: `reserve→commit→consume` with atomic CAS and idempotent consume; a consumed cap or an old proof fails `ConsumedCap`, and task/agent binding blocks cross-task/cross-agent reuse. **Backing:** Capability Replay Rejection Rate (§08, target ≈ 100%); adaptive attacks #3/#5/#6; mechanism suite double-spend = 0.

### Q15. "Can shell bypass canonicalization?"
For arbitrary shell, **yes — and we do not claim otherwise**: `command` is undecidable to canonicalize. So `run_shell` accepts only **structured allowlisted templates** (typed holes + `allowed_env`/`allowed_cwd`/`stdin_policy`); `sh -c`/pipes/redirects/base64/network and out-of-contract env/cwd/stdin are denied or routed to endorsement. **Backing:** `CANONICALIZATION_BYPASS_GATE.md` shell rows (#22–#31) and template-argument-injection (#29); the non-soundness for arbitrary shell is stated in §06/§11.

### Q16. "Why was AgentArmor not a main baseline?"
AgentArmor is **real, relevant, and cited** (§12): it abstracts the agent runtime **trace** into CFG/DFG/PDG and uses a graph annotator + inspector/type system to detect unsafe data/control flow. We include an **AgentArmor-related configuration as an auxiliary baseline on a subset** (§09), not a main one, for two principled reasons. (1) **Different point in the pipeline:** AgentArmor reasons over a (post-hoc or running) trace to *detect* unsafe flow; CapProof acts **before execution**, requiring a high-impact action to *consume* a capability rooted in trusted intent — so a head-to-head on raw ASR compares a detector to an enforcer and is easy to make misleading in either direction. (2) **Faithful reproduction risk:** a trace-graph type system is sensitive to trace construction and annotation choices; an uncalibrated reimplementation could under- or over-state its strength, so we run it representatively, label it as such, and **do not claim to beat it**. It belongs in **trace-analysis related work** and as a breadth comparator, which is exactly where we place it. **Backing:** §12 delta row (trace-time type-checking vs pre-execution consumption); §09 auxiliary-baseline policy.

### Q17. "Why not make the proof object optional?"
**We are prepared to — that is exactly what `ProofObjectRemoved` (§09.4) decides, and the design already treats the proof as separable from the identity.** CapProof's identity is **capability-consuming authorization**: a high-impact action must *consume a scoped capability* and the deterministic monitor must *verify the authorization chain*. The carried proof DAG is one **encoding** of that verification (agent emits a witness; monitor checks it) versus the monitor re-deriving it by searching store/provenance state. If the ablation shows that dropping the carried proof leaves security unchanged and does not raise latency, then the proof object is **optional by construction**: we make it an opt-in artifact whose value is confined to **auditability, replayability, failure localization, and decoupled construction** (synthesizer separate from verifier — useful when proposer and enforcer are different components/vendors), and we cite PCAA for the certificate/receipt layer. We do **not** keep a mandatory proof object if the data says it earns nothing, and because the contribution is *consumption + chain verification*, making the proof optional does not weaken the claim. **Backing:** §09.4 six-axis result; §12.4; the deep-action-only latency hypothesis (carry the proof only where derivation is deep, search otherwise) is itself a reportable design point.

### Q18. "Why not start with AuthLaunderBench instead of the kill test?"
Because a 150-task benchmark showing CapProof is low-ASR proves **coverage**, not **delta** — a reviewer would still ask "does a correctly-configured CaMeL/CLAWGUARD do the same?" The kill test (`KILL_TEST_PLAN.md`) answers the existential question first, cheaply: 10–15 head-to-head tasks against CaMeL / PACT-style / AUTHGRAPH-style / CLAWGUARD-style on the five laundering channels, with three acceptable outcomes (security gap / usability gap / proof-auditability gap) and a pre-committed pivot if none is real. Building the full benchmark **before** the kill test risks investing months instrumenting a thesis that a single strong baseline already covers. So the order is deliberate: **kill test first (Figure 1, gates everything), AuthLaunderBench second (generalizes a validated gap)**. If the kill test fails, we pivot rather than build the benchmark (`KILL_TEST_PLAN.md` §5). **Backing:** §08.2 ordering; the benchmark is explicitly "built after the kill test passes" (§00, §07).

### Cross-cutting posture
- Every "did you consider X" → a section + a number, not a promise.
- We concede the real soft spots openly (exfiltration vs CaMeL; dynamic-task endorsement burden; arbitrary shell; truly-unobserved adapter fields) and show each is bounded/measured.
- We do not defend the "proof-carrying" brand if `ProofObjectRemoved` says it adds nothing — pre-committing defuses Q1/Q11/Q17.
